# 每日论文推荐

这是面向用户的一句话入口。对用户来说，正常只需要说一次：

- `今日论文推荐`
- `过去3天论文推荐`
- `过去一周论文推荐`

## 执行原则

1. 先识别时间范围：
   - `今日论文推荐`、`每日推荐`、`今日论文` -> 当天
   - `过去3天论文推荐`、`最近3天论文` -> 3 天
   - `过去一周论文推荐`、`看看这周有啥论文` -> 7 天
2. 自动调用 `/daily-papers-fetch`。
3. 第 1 步完成后，自动调用 `/daily-papers-review`。
4. 第 2 步完成后，自动调用 `/daily-papers-notes`。
5. 全部完成后，用一句话告诉用户：
   - 推荐文件已生成
   - 重点论文笔记已生成多少篇
   - 目录页是否已自动刷新

## 失败处理

- 如果时间范围不明确，默认按 3 天处理，并在开始时说明采用的范围。
- 如果 `/daily-papers-fetch` 失败，停止流水线，报告抓取失败原因和输出文件状态，不继续点评或生成笔记。
- 如果 `/daily-papers-review` 失败，保留抓取结果，停止笔记生成，并说明推荐文件缺失。
- 如果 `/daily-papers-notes` 部分失败，报告成功生成的笔记数、失败论文列表和目录页是否刷新。

## CHECKPOINTS

- STOP：当用户要求超过 7 天范围时，先确认是否继续；长时间范围会增加抓取和笔记生成成本。
- STOP：当用户只想跑抓取、点评或笔记中的某一步时，不自动跑完整流水线，交给对应 skill。

## 重要约束

- 不要先要求用户手动跑 `跑一下论文抓取 / 点评 / 笔记`。
- 这 3 句是内部流水线和调试入口，不是首页主交互。
- 如果用户明确只想跑其中一步，再交给对应 skill。
- 不要在抓取失败后编造推荐结果。
- 不要把内部调试命令暴露成用户必须手动执行的步骤。

## 自动化

- 本 skill 本身就是“一步跑完整流水线”的入口。
- 如果用户想做本地定时任务，默认也应该触发这一句，而不是写死三条内部命令。
